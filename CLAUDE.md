# お絵描き＆塗り絵アプリ（子ども向けPWA）

## このプロジェクトについて

iPad mini で子どもが直感的に遊べる、お絵描き＆塗り絵の Web アプリ（PWA）。
Safari で開いて「ホーム画面に追加」して全画面で使う。App Store 審査は通さない。

- ターゲット：子どもが迷わず遊べることを最優先
- 対象デバイス：iPad mini（Safari）
- 提供形式：静的な HTML/CSS/JS のみ。ビルドツールやフレームワークは使わない
- 公開：GitHub Pages / Netlify などに置く（同一オリジンで配信して Tainted Canvas を回避）

## 開発の進め方

- **いきなり全部作らない。** まず最小プロトタイプから。
- 最小プロトタイプの目標：**1枚のSVG線画に対して、境界を尊重したバケツ塗りがちゃんと効くこと。** これだけ。
- ペン・色選び・複数SVG切り替え・保存などは、バケツが動いてから後回しで追加する。
- 設計判断や不明点があれば、コードを書き始める前に必ず確認すること。

## 最重要：バケツ（塗りつぶし）の設計

ここがこのアプリで一番壊れやすい所。以下を厳守すること。

- キャンバスは**視覚的に2層**に重ねる：上＝SVG線画、下＝色。塗っても線画が消えないようにする。
- **色レイヤー単体で Flood Fill してはいけない。** 色レイヤーには壁（線）が無いので、全面に色が漏れる。
- 正しい方式：**色は下の色レイヤーに書き込むが、境界の判定は線画レイヤーのピクセルを「壁」として読む。**「塗る対象」と「境界を判定する対象」を分けて考える。
- **アンチエイリアス対策：** SVGをラスタライズすると線のフチが半透明グレーになる。単純な色一致のバケツだと「線の内側に白いフチが残る」か「隙間から漏れる」のどちらかが必ず起きる。線画を境界マスクとして扱ってこれを回避する。
- **Flood Fill は再帰で書かない。** iPad全面は数百万ピクセルあり、再帰だとスタックが溢れる。キュー方式かスキャンライン方式で実装する。

## その他の必須対応

- **devicePixelRatio 対応を最初から入れる。** iPad は高解像度なので、放置するとタップ座標とキャンバスの実ピクセルがズレて「押した所と違う場所が塗られる」＋描画がぼやける。後から入れると座標系の作り直しになるので初手で。
- **Safari の封じ込め：**
  - `touch-action: none`
  - ピンチズーム無効化
  - pull-to-refresh（引っ張って更新）の無効化
  - `apple-mobile-web-app-capable` 系の meta タグ
  - 子どもの誤操作でスクロール・拡大・更新が暴発しないようにする
- **SVGの扱い：** `<img src>` 経由で canvas に描くより、SVG を HTML にインライン展開してから使う方が taint 事故が起きにくく、境界判定もやりやすい。
- **画像はローカル保存：** 塗り絵のSVGは Google ドライブ等の外部から読み込まず、アプリと同じフォルダ内（`images/`）に置く。外部読み込みは Tainted Canvas に引っかかりバケツが機能しなくなる。

## UI方針

- 子どもが迷わないよう、ボタンは大きく・少なく。
- 色は最初は基本の数色（赤・青・黄・緑など）を固定ボタンで用意。色数を増やしすぎない。
- 押し間違いしにくい配置。iPad mini の画面サイズを前提にする。

## 想定ファイル構成

```
oekaki-app/
├── index.html   アプリ本体（画面枠とキャンバス）
├── style.css    デザイン（iPad向けボタン配置・全画面設定）
├── main.js      ロジック（ペン/バケツ切替、塗る仕組み）
└── images/      塗り絵SVGの置き場
    ├── car.svg
    └── star.svg
```

## やらないこと

- フレームワーク・ビルドツール・npm パッケージの導入
- 外部サーバーや API との通信
- 外部URLからの画像読み込み
